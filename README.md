## What is PartiBench?

Deploying a neural network across heterogeneous IoT–Edge–Cloud nodes requires answering two questions: *how* should the model be split, and *where* should each piece run? The answer depends heavily on the specific hardware and network conditions of the deployment, there is no universal best strategy.

PartiBench provides the tooling to answer these questions empirically. It consists of two independent systems that are used in sequence:

**1 PartiBench (benchmarking tool)**

Runs locally on each node in your infrastructure. It profiles every layer block of VGG16 under all partitioning strategies and partition sizes, measuring:
- execution time (mean, median, std over 5 post-warm-up runs)
- memory footprint
- input and output tensor sizes

The output is a structured JSON dataset (one file per node) that captures how that specific hardware performs on each piece of the model. These files are then merged into a single `benchmark.json` that covers the entire infrastructure.

**2 Distributed CNN Inference Orchestrator**

Takes a `benchmark.json` and a user-defined placement strategy (`hs.txt`) and actually deploys the model across your nodes. Each participating node runs a lightweight daemon that receives its component assignment and spawns the appropriate processes. The result is a live distributed inference pipeline that measures real end-to-end latency.

The typical workflow is: profile first with PartiBench → inspect the data to choose a placement strategy → deploy with the orchestrator → measure end-to-end latency → iterate.

## Concepts

### VGG16 layer blocks

VGG16 is treated as a flat sequence of **16 layer blocks** (blocks 0–15). Each block is a contiguous group of layers (convolution, activation, pooling, or fully-connected) and is the atomic unit of partitioning. Block boundaries are derived automatically from the model by `benchmark.py`.

### Partitioning strategies

| Strategy | How it works | When it helps |
|---|---|---|
| **No split** | One node runs the full block | Simple baseline; no coordination overhead |
| **Pipelined** | Different sequential block segments assigned to different nodes | Reduces communication to a single intermediate tensor; good when one node is too slow alone |
| **Input split** (data partitioning) | Multiple nodes each process a different spatial slice of the input through the full block | Parallelises computation; effective when compute is the bottleneck and nodes are near each other |
| **Filter split** (parallelised model partitioning) | Multiple nodes each hold a different subset of filters and process the full input | Reduces per-node model memory; useful in mid-convolutional layers under memory pressure |

Consecutive input-split blocks chained with `"merger": null` form **deep input partitioning**: each node processes its slice through several layers before outputs are merged, amortising the merge cost.

### Placement strategy file (`hs.txt`)

`hs.txt` defines the distributed deployment. It has **exactly 16 lines**, one per VGG16 block:

```json
{"split_method": "<method>", "mapping": {<node>: <ratio>, ...}, "merger": "<node>" | null}
```

**`merger`** is a key field as it is a **chain flush signal**, not a per-block output address:
- `null` — chain this block into the same BLOCK process as the next line. No data transfer yet.
- `"node_name"` — end the current chain; a BRIDGE is created at this node to collect, merge, and forward outputs to the next chain. **Network transfer only happens here.**

When `place.py` reads the file it groups consecutive `null`-merger lines into a single BLOCK process. The non-null merger line ends the group and triggers inter-node communication.

#### Example: pipelined partitioning (the default `hs.txt`)

```
{"split_method": "no_split", "mapping": {"node_A": 1}, "merger": null}    ← blocks 0–5: chain on node_A
...
{"split_method": "no_split", "mapping": {"node_A": 1}, "merger": "node_A"} ← block 6: flush → ~0.8 MB sent to node_B
{"split_method": "no_split", "mapping": {"node_B": 1}, "merger": null}    ← blocks 7–14: chain on node_B
...
{"split_method": "no_split", "mapping": {"node_B": 1}, "merger": "node_B"} ← block 15: flush → predictions returned
```

#### Example: deep input partitioning across 5 nodes (blocks 0–6)

```json
{"split_method": "input_split", "mapping": {"iot_A1": 0.12, "iot_A2": 0.11, "iot_A3": 0.10, "edge_A": 0.26, "cloud": 0.41}, "merger": null}
... (same mapping for lines 2–6)
{"split_method": "input_split", "mapping": {"iot_A1": 0.12, ...}, "merger": "cloud"}
{"split_method": "no_split",    "mapping": {"cloud": 1}, "merger": null}
... (lines 8–15)
{"split_method": "no_split",    "mapping": {"cloud": 1}, "merger": "cloud"}
```

> Input split ratios must match values available in the benchmark (`0.25`, `0.5`, `0.75`, `1.0`) and must sum to 1.

### Component model

`place.py` translates `hs.txt` into three types of OS processes, spawned on each node by `local_deploy.py`:

| Component | Where it runs | What it does |
|---|---|---|
| **PRODUCER** | Start node | Loads `data/ship.png`, sends it into the pipeline, receives predictions, measures end-to-end time |
| **BLOCK** | Any node | Receives an input tensor, runs its assigned VGG16 layers, sends output to the next BRIDGE |
| **BRIDGE** | Merger node | Collects partial outputs from one or more BLOCKs, merges them, optionally re-partitions, and forwards to the next stage. The final BRIDGE applies softmax and returns results to the PRODUCER |

Network delays are **simulated at the application level** using the delay values from `net_rules.tsv`, so the system can be tested on a single machine without actual bandwidth constraints.

## Dependencies

All Python files target **Python 3**. MXNet 1.9.1 requires NumPy < 1.24, so a virtual environment is strongly recommended:

```bash
python3 -m venv partibench_env
source partibench_env/bin/activate
pip install "numpy<1.24" mxnet gluoncv networkx psutil matplotlib
```

Shell scripts (`configure.sh`, `shape.sh`) require Linux with `tc` (iproute2) and optionally cgroup v2 for CPU throttling. They must be run as root. VGG16 weights (~500 MB) are downloaded automatically by GluonCV on first use.

## Local 2-Node Example

The fastest way to run the full pipeline end-to-end. `local_example/` simulates two nodes on a single machine using loopback addresses:

| Virtual node | IP | Role |
|---|---|---|
| `node_A` | 127.0.0.1 | Runs VGG16 blocks 0–6 |
| `node_B` | 127.0.0.2 | Runs VGG16 blocks 7–15; returns predictions |

All commands below are run from `local_example/` with the virtual environment active.

### Step 1 — Generate configuration

```bash
cd local_example
python setup_local.py
```

Produces `cc_graph.gml`, `net_rules/`, and `hs.txt` for the 2-node pipelined setup.

### Step 2 — Run PartiBench

Profiles all 16 VGG16 blocks locally (~7 min on a typical machine):

```bash
cp net_rules/node_A_rules.tsv net_rules.tsv
python ../benchmarking/benchmark.py node_A vgg16 True
```

Output: `node_A.json` - execution times, memory, and tensor sizes for all blocks and partition configurations.

### Step 3 — Merge benchmark data

```bash
cp node_A.json benchmarks/nodes/node_A.json
python ../benchmarking/merge_bench.py
```

Output: `benchmarks/benchmark.json` - the unified dataset used by the orchestrator and the plotting script.

### Step 4 — Deploy and run inference

Open **two terminals** (one per virtual node), activate the environment in each, and start a deployment agent in each:

```bash
# Terminal 1 - node_B
python ../orchestrator/local_deploy.py --ip 127.0.0.2

# Terminal 2 - node_A
python ../orchestrator/local_deploy.py --ip 127.0.0.1
```

Both print `Waiting for component deployment info...` and block.

In a **third terminal**, run the orchestrator to send component assignments:

```bash
python ../orchestrator/place.py \
    --start-node node_A \
    --graph cc_graph.gml \
    --bench benchmarks/benchmark.json \
    --hs hs.txt
```

Switch to **node_A's terminal** and type `s` to start inference. You will see:

```
PRODUCER: Inference took: XXXX ms
PRODUCER: Results:
container ship, container vessel: 84.32%
wreck: 5.17%
lifeboat: 3.91%
```

The system repeats automatically every 5 seconds for 15 more runs. Type `r` instead of `s` to re-deploy with a different `hs.txt` without restarting the agents.

## Multi-Node Deployment (EC2 / Real Hardware)

This is how the paper's experiments were conducted. The setup uses 5 nodes (3 IoT, 1 Edge, 1 Cloud) on Amazon EC2, with CPU throttling and Linux `tc` to emulate realistic CC conditions.

### EC2 node configuration

| Node | Role | Memory | CPU |
|---|---|---|---|
| iot\_A1, iot\_A2, iot\_A3 | IoT | 536–1073 MB | 33% × 1 vCPU @ 2.5 GHz |
| edge\_A | Edge | 2147 MB | 100% × 2 vCPUs |
| cloud | Cloud | 17179 MB | 100% × 4 vCPUs |

Network links (one-way): IoT–IoT 200 Mbps / 8 ms · IoT–Edge 150 Mbps / 11 ms · Edge–Cloud 100 Mbps / 50 ms.

### Step 1 — Build the CC graph (once, on any node)

```bash
mkdir -p orchestrator/net_rules
cd orchestrator/
python create_graph.py
```

Produces `cc_graph.gml` and `net_rules/<node>_rules.tsv` for each node. Copy the relevant TSV to each physical node:

```bash
scp net_rules/iot_A1_rules.tsv user@iot_A1:/path/to/partibench/net_rules.tsv
scp net_rules/edge_A_rules.tsv user@edge_A:/path/to/partibench/net_rules.tsv
# ... etc.
```

### Step 2 — Configure each node (requires root)

Run on every participating node before benchmarking:

```bash
chmod +x config/configure.sh config/shape.sh
sudo config/configure.sh 33    # 33% CPU throttle for IoT nodes
sudo config/configure.sh       # no throttle for Edge/Cloud
```

This sets up the cgroup CPU limit, raises socket buffer sizes (required for VGG16 block 1's ~12 MB tensor), and applies `tc` bandwidth shaping from `net_rules.tsv`.

### Step 3 — Run PartiBench on each node

Run independently and in parallel on each node:

```bash
# On each node  run from the partibench/ directory
cd benchmarking/
python benchmark.py iot_A1 vgg16 True    # one node: full profiling (I/O sizes + memory)
python benchmark.py iot_A2 vgg16 False   # other nodes: execution times only
python benchmark.py edge_A vgg16 False
python benchmark.py cloud  vgg16 False
```

Each configuration is run 6 times; the first is a warm-up and discarded. Expected runtime: ~3 min on a powerful node, ~15 min on a constrained IoT node.

### Step 4 — Merge benchmark results

Collect all output files onto one machine and merge:

```bash
mkdir -p benchmarks/nodes/
scp user@iot_A1:/path/to/iot_A1.json benchmarks/nodes/
scp user@edge_A:/path/to/edge_A.json benchmarks/nodes/
# ... etc.

cd benchmarking/
python merge_bench.py
```

Output: `benchmarks/benchmark.json` - the unified profiling dataset.

### Step 5 — Define your placement strategy

Edit `orchestrator/hs.txt` to specify how the model should be partitioned and which nodes run which blocks. See the [Placement Strategy File](#placement-strategy-file-hstxt) section for the format and examples.

To use the paper's pipelined strategy (iot\_A1 runs blocks 0–6, edge\_A runs blocks 7–15), the file should read:

```
{"split_method": "no_split", "mapping": {"iot_A1": 1}, "merger": null}
... (repeat for lines 2–6)
{"split_method": "no_split", "mapping": {"iot_A1": 1}, "merger": "iot_A1"}
{"split_method": "no_split", "mapping": {"edge_A": 1}, "merger": null}
... (repeat for lines 8–15)
{"split_method": "no_split", "mapping": {"edge_A": 1}, "merger": "edge_A"}
```

### Step 6 — Start deployment agents on all nodes

Run on each participating node:

```bash
python orchestrator/local_deploy.py
```

Each agent prints `Waiting for component deployment info...` and blocks.

### Step 7 — Run the orchestrator (from any node)

```bash
python orchestrator/place.py \
    --start-node iot_A1 \
    --graph cc_graph.gml \
    --bench benchmarks/benchmark.json \
    --hs hs.txt
```

The orchestrator resolves the strategy, computes input-split offsets where needed, and sends each node its assignments over TCP port 6999.

### Step 8 — Trigger inference

On the start node's terminal, type `s` to begin. The PRODUCER sends `data/ship.png` through the pipeline and prints end-to-end time and top-3 predictions, repeated 16 times.

## Plotting Results

All figures from the paper can be generated with a single script (it uses data from past measurements):

```bash
python results/plot_paper_figures.py
```

Output PDFs are written to `results/PLOTS/reproduced/`.

### Plotting the pre-collected EC2 results (no experiments needed)

The raw data from all paper experiments is bundled in `results/`:

| File | Contents |
|---|---|
| `results/exp_analytical_data.txt` | Per-node execution times; memory and execution tables for filter/input partitioning at F=25/50/75%; benchmarking runtimes |
| `results/experiments.txt` | Lab notebook: all recorded runs, TCP overhead observations, serialization cost analysis |
| `results/PLOTS/final/` | Original figures |

Running the script without any arguments uses these bundled measurements

Plotting results from your own experiments:  After running PartiBench and merging results into `benchmark.json`, pass it to the script:

```bash
python results/plot_paper_figures.py --bench path/to/benchmark.json
```

To regenerate only specific figures:

```bash
python results/plot_paper_figures.py --bench path/to/benchmark.json --figs 6 8 10
```

To save figures to a custom directory:

```bash
python results/plot_paper_figures.py --bench path/to/benchmark.json --out path/to/output/
```

## Notes and Known Limitations

- **Automatic placement:** `place.py` contains a `place()` function that greedily optimises placement using benchmark data which is currently disabled in favour of `load_hardcoded_strategy()`.
- **NumPy compatibility:** MXNet 1.9.1 requires `numpy < 1.24` (`np.bool` was removed in 1.24). Always use the virtual environment.
- **Ports:** Component ports are allocated sequentially from 5000. Port 6999 is the fixed deployment listener. Ensure these are reachable between CC nodes.
- **FC input split:** The BRIDGE hardcodes `relu` when merging input-split fully-connected layers. This is VGG16-specific and would need updating for other architectures.
- **MobileNetV2:** Partially implemented in `core.py` but not wired into the benchmark or orchestrator pipelines.
