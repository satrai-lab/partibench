A. Show the benefits of data partitioning (implemented using input split)

	NOTE: Make iot nodes homogeneous (33% x 1 vCPU each)

	- Since there are a lot of possible placements that employ the data partitioning strategy, it is difficult to construct an example manually when focusing on the entire CNN. Thus, we will try to isolate only a set of consecutive blocks and apply input split on them. Afterwards, we will compare the execution time of the input split partition to its centralised version.

	- How deep should input split be? (i.e. how many consecutive blocks to apply input split to?)

		Before deciding, it is important to know that required input size depends on the input split depth. For example, required input size (in bytes) for 25% input split of blocks:

			0: 153240
            0-1: 155928
            0-2: 161304
            0-3: 166680

            0-8: 241944
            0-9: 284952
            0-10: 327960
            0-11: 370968

            TODO show plot here not numbers

        That means that we cannot extend deep input split too much because: as required input size increases, communication costs increase and work against the benefit of workload parallelization.

        What if we make deep input split very small?
        The smaller the input split, the less the required input size (which is good). However, there is a problem: having a lot of small input split parts in the CNN increases communication costs by a lot since data transmission happens more frequently.

        So, we must find a balance for the optimal input split depth.

        For our case, we will choose the input split depth arbitrarily, focusing on blocks 0-6.

	- How to find a good workload (input) distribution?

		Assume data start from iot_A1 and let's try distributing workload to 5 nodes and set "cloud" node as the merger

		Let T_i be the total time needed for node i including time to transfer data from iot_A1 to i, time to execute inference on i and time to transfer output from i to merger. Find p_i, the percentage of input that each node i must be assigned so that all nodes finish at the minimum same time T = sum(T_i). 

		Ti = (exec_i + tr_i) * p_i + lat_i

		where:
			> exec_i: execution time for set of blocks on node_i
			> tr_i: total data transmission time (without latency) for starting node -> node_i and node_i -> merger
			> lat_i: total transmission latency for the above communications

		IMPORTANT: for simplicity, this analysis assumes a linear relation between execution time and input split % for a set of blocks. E.g. if execution time for blocks 0-6 with 100% input is 150 ms, then for 10% input it should be 15 ms (theoretically). In reality this is not true because of MXNet internal functionality and overheads. Observation for cloud node: execution time for 100% input -> ~150 ms and for 10% input -> ~40 ms.

		- Measure data transmission times using full inputs and outputs, EXCLUDING latencies:

B. Is parallelised model partitioning (implemented using filter split) actually beneficial?

	- To test this, analyze benchmarking results for multiple nodes by comparing filter split vs input split blocks (same percentage):

		a. Compare execution times

			- If filter split blocks are similar to or worse than input split blocks (EXPECTED), then filter split will always be worse than input split, since it needs greater input sizes (increased communication costs)
			- If filter split blocks are better than input split blocks (NOT EXPECTED), then filter split may be better than input split, when required input is small or network capacity is high (thus communication cost is not a problem)

		b. Compare memory footprint

			- If filter split blocks have smaller memory footprint than input split blocks, then filter split may be useful in cases where memory is limited (NOT EXPECTED)
			- However, if filter split blocks have greater/equal memory footprint than input split blocks, then filter split is not useful in limited memory scenarios (EXPECTED).
			- Why is the second case more likely than the first one?
				input split: full layer block + smaller input
				filter split: smaller layer block + full input

	- After checking benchmarking numbers:

		a. About execution times:

			- For most layer blocks, execution times are similar (or even worse) in filter split compared to input split. Only for a very small amount of layer blocks filter split is faster than input split. 		As a result:

		> If there is no extreme memory limitation, then data partitioning (input split) is going to be preferred almost every time, since it can achieve workload parallelization with smaller total time costs i.e. lower communication time because of reduced data transmission and similar execution time with parallelized model partitioning (filter split)

		> In a scenario where there are extreme memory limitations and at the same time there's need for privacy (i.e. keeping data as close as possible and avoid distant powerful nodes), parallelized model partitioning (filter split) may be useful. However, one should keep in mind that unless network capacity is high enough, total inference time may be adversely affected.
